SQL essentials
window functions
ROW_NUMBER() — unique rank, no ties
RANK() — ties get same rank, next rank skipped
DENSE_RANK() — ties get same rank, no gap after
LAG(col,1) — previous row value in partition
LEAD(col,1) — next row value in partition
SUM(col) OVER (ORDER BY date) — running total
PARTITION BY resets window per group; GROUP BY collapses rows
SELECT user_id, date, revenue, SUM(revenue) OVER ( PARTITION BY user_id ORDER BY date ) AS running_total FROM orders;
CTEs & subquery patterns
CTE (WITH) — named, reusable, readable. Prefer over nested subqueries
Correlated subquery — references outer query; runs once per row (slow)
Subquery in FROM — inline view, runs once (better)
EXISTS vs IN — EXISTS short-circuits; IN materializes the set
WITH monthly_rev AS ( SELECT user_id, DATE_TRUNC('month', date) AS mo, SUM(revenue) AS rev FROM orders GROUP BY 1, 2 ) SELECT *, LAG(rev) OVER (PARTITION BY user_id ORDER BY mo) FROM monthly_rev;
join quick ref
INNER JOIN — only matching rows in both tables
LEFT JOIN — all left rows, NULLs for no match on right
FULL OUTER — all rows from both, NULLs where no match
SELF JOIN — table joined to itself; needs alias
Use JOIN when — combining related tables; use subquery when filtering by aggregation
conditional aggregation
Count only rows meeting a condition without a subquery
SELECT COUNT(CASE WHEN status='paid' THEN 1 END) AS paid, COUNT(CASE WHEN status='refund' THEN 1 END) AS refunds, SUM(CASE WHEN status='paid' THEN amount ELSE 0 END) AS revenue FROM transactions;

pandas core patterns
df.groupby('col').agg({'val':'sum'}) — group + aggregate

df['col'].fillna(df['col'].median()) — fill missing

EDA checklist
1df.shape, df.info(), df.describe() — understand structure
2df.isnull().sum() — check missing values
3df.duplicated().sum() — check duplicates
4df['col'].value_counts() — distribution of categoricals
5Check date range, outliers with describe() percentiles
6Correlation matrix for numeric columns
7State findings + hypotheses before modeling
A/B testing framework
experiment design steps
1Define hypothesis — what change, what expected effect
2Choose primary metric + guardrail metrics
3Calculate required sample size (power analysis)
4Randomize users into control / treatment groups
5Run experiment until sample size reached
6Check p-value against α=0.05; check CI does not cross 0
7Check for novelty effect, segment breakdowns
stats quick ref
p-value — probability of seeing this result if H₀ is true. p < 0.05 → reject H₀
Type I error (α) — false positive: reject H₀ when true. Usually 5%
Type II error (β) — false negative: fail to reject H₀ when false
Power (1−β) — probability of detecting a true effect. Target 80%
CI — if 95% CI excludes 0, result is significant at α=0.05
p-hacking — peeking at results early inflates false positive rate
Metrics & business sense
metric drop diagnosis framework
1Confirm the drop is real — check data pipeline, logging issues
2Scope it — which segment? platform, region, user cohort?
3Timeline — when did it start? Any deploys, campaigns, holidays?
4Funnel breakdown — where in the funnel is the drop?
5External factors — competitor, seasonality, macro?
6Propose hypothesis + next action
key metric definitions
DAU/MAU ratio — stickiness. Higher = more engaged users
Retention D7/D30 — % users who returned 7/30 days after first use
LTV — avg revenue per user over their lifetime
CAC — cost to acquire one customer. LTV/CAC > 3 is healthy
Conversion rate — % users completing desired action in funnel
North Star — single metric that best captures delivered value
Data modeling reference
star schema vs snowflake
Star schema — fact table + denormalized dimension tables. Fast queries, simple joins
Snowflake schema — dimensions further normalized. Less redundancy, more joins
Fact table — measures/events (revenue, clicks). Foreign keys to dims
Dimension table — descriptive attributes (user, product, date)
Grain — the level of detail one row represents. Define first
Slowly Changing Dim — Type 1 (overwrite), Type 2 (history rows)
modern data stack
Ingestion — Fivetran, Airbyte: pull source data into warehouse
Warehouse — BigQuery / Snowflake / Redshift: store + compute
Transform — dbt: SQL-based modeling, tests, documentation
BI layer — Tableau / Looker / Power BI: dashboards for stakeholders
Partitioning — divide table by date/region to limit scan cost
Materialized view — pre-computed query result; refreshed on schedule
